# M4 长任务能力评测用例

## 目标

验证 Agent 在多轮、长上下文和跨进程场景中仍能保留计划、恢复会话、加载长期记忆，并按照持久化任务依赖继续工作。

## 用例 1：Todo 持续更新

提示词：

```text
阅读一个包含多个模块的项目，列出实现、测试、文档三个步骤，完成每一步后更新状态。
```

通过标准：

- 多步骤工作开始前调用 `todo_write`。
- 每次调用以完整列表替换当前 Todo，状态只使用 `pending`、`in_progress`、`completed`。
- 最终没有遗漏的 `pending` 项。

## 用例 2：Session 跨进程恢复

第一次运行使用 `--session m4-demo`，要求 Agent 检查项目并给出后续修改计划；退出后用同一 Session ID 再次运行并要求继续。

通过标准：

- 第二次运行能读到第一次的消息、工具调用结果和 Todo。
- 恢复后 system prompt 只保留一份，新的消息只追加一次。
- Session 文件损坏时明确报错，不静默覆盖历史。

## 用例 3：上下文压缩

让 Agent 连续读取多个大文件并执行产生较长输出的命令，再继续完成原始目标。

通过标准：

- 压缩顺序为大结果落盘、消息裁剪、旧工具结果占位、超阈值摘要。
- `assistant.tool_calls` 与对应的 `tool` 消息不会被拆开。
- 完整历史先写入 transcript；压缩后仍保留当前目标、关键决定、修改文件和剩余工作。
- API 返回 context/prompt too long 时只进行有界的 reactive compact。

## 用例 4：跨会话 Memory

第一轮要求 Agent 记住项目约束，例如“测试不得访问真实外部服务”；新建会话后要求实现一项测试。

通过标准：

- 约束被写入 `.memory/`，并出现在稳定的 `MEMORY.md` 索引中。
- 新会话能按查询加载相关记忆，不注入无关记忆。
- Memory 在 Session 压缩后仍然存在。

## 用例 5：持久化任务图

创建三个任务：数据库 schema、API（依赖 schema）、测试（依赖 API），然后尝试按逆序认领。

通过标准：

- 被未完成或缺失依赖阻塞的任务不能认领。
- 合法状态机是 `pending -> in_progress -> completed`。
- 完成上游任务后，下游任务被解锁；重新启动进程后状态保持不变。
- 自依赖或依赖环不会写入部分状态。

## 建议记录

- 总 LLM 调用次数与工具调用次数
- 压缩前后估算 token/字符数
- transcript 和大工具输出落盘次数
- Session 恢复成功率
- Memory 命中数与无关注入数
- Task 阻塞、认领和解锁结果
